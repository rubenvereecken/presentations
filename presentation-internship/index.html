5. Ontwikkelomgeving en tools

Vertellen over verschillende tools aangezien er niet te veel gezegd kan worden over het project zelf. Wel een belangrijk deel van mijn leerervaring, dus best de moeite om iets over te zeggen. Gezien ik niet zoveel over het project mag prijsgeven wil ik graag een aantal tools en werkwijzen toelichten die ik heb opgestoken. Voor mij betekenen ze een belangrijk deel van wat ik tijdens mijn stage heb geleerd.

Version Control

Perforce is het standaardsysteem:

  • Iedereen interageert met centrale server
  • Iedereen heeft gedeeltelijke checkout, maar toch veel dependencies

Maar ook veel andere flavors om met code te werken.

  • Client in the Cloud (CitC)
  • Git5: git wrapper
Reeds veel ervaring met Git, maar nu andere dingen geleerd. In Google wint gewoon het systeem dat het beste is, er zijn veel manieren om met code om te gaan. Ik gebruikte git5 en heb eigenlijk nooit perforce gebruikt. Het heet git5 omdat de vorige tool g4 heette. God knows.

Code Search en Critique

Code Search

  • Platform om in de code te zoeken

Critique

  • Platform om met ChangeLists te werken
  • Gebruikt voor het review proces
  • Motivaties voor code
Het meeste is al eens geprobeerd in Google, dus Code Search is de goto als je iets nieuws wil doen. Code Search is nauw gelinkt aan Critique, het platform om aan code reviews te doen. Ik heb de link vaak gebruikt toen ik zocht _waarom_ iemand nu net zoiets zou doen.

Code Reviews

  • Controle op kwaliteit, tests,...
  • Owner approval
  • Readability per taal
Loopt mooi over van Critique. Anekdote: Op een bepaald moment had een intern een review van iedereen met JS readability gevraagd. Hilarity ensued en de intern heeft zijn code er nooit in gekregen. Nog nooit is iets zo grondig nagekeken geweest.

Bazel

  • Google's build tool a la Make
  • Volledig gedistribueerd, niets wordt lokaal gecompileerd
  • Werkt voor alle talen, eenvoudig uitbreidbaar
  • Automatische dependency management
  • Automatisch tests bouwen en uitvoeren

Protocol Buffers

  • Een Google-wijd dataformaat en meer
  • Automatisch gegenereerde compatibele clients en servers
  • Staat toe dat protocols uitgebreid worden
  • Ook gebruikt voor RPC services
  • Enorm performant in binair formaat

6. Mijn Project: Autodash

The ultimate Automation Dashboard.

Ik heb geluk gehad het begin van Autodash mee te maken.
Daardoor mocht ik het volledig mee designen, had enorm veel input

Achtergrond

  • Automation: automatisering van processen zoals rollouts
  • Maar... Een strijd gestreden op vele fronten
  • Vele pogingen om een universeel platform te maken
  • Huidige stand: iedereen heeft eigen flavor van laatste poging

Automatisering van rollouts: vermeld canary, feature testing, verhuizen van cluster

Elk team lost zijn eigen problemen op. De basis is grotendeels uniform: clusterbeheer.

Grote figuren die Google SRE scene goed kennen strijden voor een convergerende beweging, in plaats van divergerend zoals dit moment het geval is. Mijn baas is een van die frontmannen.

Er is al heel veel geld gepompt in oplossingen. De laatste, die door veel mensen wordt gebruikt, voldeed voor niemand volledig. Iedereen heeft zijn eigen versie gemaakt en moet die dan ook zelf hosten. Daardoor is er voor elk team een eigen flavor. Dit platform met zijn vele flavors is dan ook het hoofddoelwit voor ons.

Wat is Autodash

  • Automation Dashboard
  • Het begin van een oplossing voor de huidige dystopie
  • Een dashboard dat een snel overzicht biedt
  • ... En daarbij ook heel geavanceerd zoekbaar voor specifieke info
  • Haalt informatie uit alle relevante bronnen
  • Federated: wordt aangeboden als Platform As A Service
Ik kan niet teveel details vertellen. Beschouw Autodash als aggregator van informatie die die informatie dan beschikbaar stelt en heel efficient zoekbaar maakt. Sommige teams hebben wel 5 verschillende platforms waar ze naar moeten kijken. In geval van nood moet het snel gaan. Autodash is de perfecte oplossing voor deze teams. Doel: sympathie winnen. Eenvoudig beginnen, daarna meer en meer kracht geven. Uiteindelijk worden oude platforms vervangen.

Voorbeeld van info: Hoever staan mijn rollouts, zijn er ergens errors.

Autodash Design

  • De basis van alles is beseffen dat het gaat om informatie verzamelen en displayen
  • De Sources of truth zijn de bronnen die informatie pushen
  • Informatie kan gestuurd worden via het public protocol, naar de indexer
  • Er is maar 1 indexer. Schaalt slecht.
  • Er zijn wel een hoop frontends, zodat we daar een hele load kunnen verdragen
  • Veel gediscussieerd om te cachen in frontends omdat die gemakkelijk scalen
  • Maar nog niet, gezien sommige info heel tijdsgevoelig is, hoewel meeste niet
  • Frontend servers hielden wat info bij in een db, gebruiker specifiek
  • Frontend was configureerbaar met config bestanden, die regelmatig herladen werden
  • Zo konden we gemakkelijk veranderingen pushen
  • Waarom geen meerdere indexers? Index was in-memory, nog niet zo goed uitgedacht

Autodash Implementation: Frontend

  • Server geschreven in Go
  • One-page web app in Angular.js 1.3
  • Alle data over JSON requests
  • Geschreven in JavaScript Closure
  • Google Material's design specification
  • RPC's werden doorgestuurd naar Indexer

Closure: Dialect van JS. Compileerbaar dus statically typed, ook minified en obscured. Alternatief zou TypeScript zijn wat dus geen JS meer is maar compileert naar JS. Closure was een verschrikkelijke ervaring, met een paar interne bugs etc. Meer gevecht dan winst.

Autodash Implementation: Indexer

  • Geschreven in Go
  • Meeste data in-memory bijgehouden
  • De rest in een gedistribueerde NoSQL database
  • Endpoints voor de public API om data aan te nemen
  • Endpoints voor de interne API om queries te beantwoorden

Autodash Resultaten

  • Volledig bruikbaar prototype dat met echte data werkt
  • Goed ontvangen op de Automation Summit
  • Reeds een aantal experimentele gebruikers die nieuwsgierig waren
  • Grondig gedekt door tests
  • Volledig gemonitord op gebruik

Maar...

  • Nog niet schaalbaar voor veel data
  • Problemen met 'verse' data
  • Nog niet erg veilig
